Многопоточность в Java

Race condition и четыре способа его победить — на примере торгового стакана.

1. Зачем потоки вообще

Две причины:

  1. Отзывчивость UI. HTTP-запрос на 3 секунды в UI-потоке замораживает окно.
  2. Использовать все ядра CPU. Современный ноутбук — 8 ядер.

В этой методичке мы фокусируемся на проблемах, которые возникают при общем изменяемом состоянии.

2. Race condition: корень зла

В пуле 10 акций. Двое клиентов одновременно хотят купить по 7.

int available = 10;

public boolean buy(int qty) {
    if (available >= qty) {   // (1) проверка
        available -= qty;      // (2) действие
        return true;
    }
    return false;
}

В однопоточном мире — работает. В многопоточном — ломается:

ВремяThread AThread B
t=1читает available=10
t=2проверка 10>=7 ok
t=3читает available=10
t=4проверка 10>=7 ok
t=5available -= 7 → 3
t=6available -= 7 → -4 !!!

Оба клиента прошли проверку, оба купили. Продано 14 акций из 10. Биржа должна доставить несуществующее — баг.

Почему available -= 7 не атомарная:
  1. прочитать available в регистр,
  2. вычесть 7,
  3. записать обратно в память.
Между ними может вклиниться другой поток.

3. Видимость vs атомарность

  • Атомарность. Операция прервана посередине. Наш race condition.
  • Видимость. Поток A записал флаг, поток B его не видит никогда — JIT закешировал чтение в регистре.

Все 4 решения ниже решают обе проблемы сразу — volatile отдельно не нужен.

4. Решение 1: synchronized

public synchronized boolean buy(int qty) {
    if (available >= qty) {
        available -= qty;
        return true;
    }
    return false;
}

У каждого объекта есть монитор. synchronized на методе = захватить монитор this на время вызова.

Плюсы/минусы

  • Просто. Нельзя забыть отпустить.
  • Решает и атомарность, и видимость.
  • Блокировка на весь метод — падает параллелизм.
  • Нельзя прервать ожидание, нельзя с таймаутом.
Правило: никогда не синхронизируйтесь на String или boxed Integer — JVM может их интернировать.

5. Решение 2: ReentrantLock

private final ReentrantLock lock = new ReentrantLock(true);

public boolean buy(int qty) {
    lock.lock();
    try {
        if (available >= qty) { available -= qty; return true; }
        return false;
    } finally {
        lock.unlock();
    }
}

Чем лучше synchronized

  • Таймаут: tryLock(500, TimeUnit.MILLISECONDS).
  • Прерываемость: lockInterruptibly().
  • Fairness: new ReentrantLock(true) — FIFO.
  • Можно держать lock через границы методов.

Чем хуже

  • Нельзя забыть unlock. Всегда try/finally.
  • Многословнее.

6. Решение 3: AtomicInteger + CAS

CAS (Compare-And-Swap) — атомарная процессорная инструкция: если текущее значение равно expected — записать newValue и вернуть true. За одну инструкцию.

private final AtomicInteger available = new AtomicInteger(10);

public boolean buy(int qty) {
    while (true) {
        int current = available.get();
        if (current < qty) return false;
        int next = current - qty;
        if (available.compareAndSet(current, next)) return true;
        // CAS не прошёл. Повтор.
    }
}

Это lock-free. Ни один поток не усыпляется.

Подводные камни:
  • ABA-проблема: значение A-B-A. CAS не заметит.
  • Одно поле. Для двух — либо AtomicReference с immutable состоянием, либо lock.
  • Голодание при высокой контенции: CPU крутит цикл.

7. Решение 4: Semaphore

Семафор — счётчик разрешений. acquire(n) забрать n, release(n) вернуть.

private final Semaphore permits = new Semaphore(10, true);
private final AtomicInteger sold = new AtomicInteger();

public boolean buy(int qty) {
    if (!permits.tryAcquire(qty)) return false;
    sold.addAndGet(qty);
    return true;
}

Классическое применение: ограничение параллельных запросов к API, пул соединений, барьер «не больше N загрузок».

Важно: семафор синхронизирует только квоту. Для других полей (sold) нужно отдельно — либо lock, либо Atomic.

9. Пять правил

  1. Не делитесь мутабельным состоянием без синхронизации. Самый простой путь — immutable.
  2. Атомарная операция начинается и заканчивается в одной критической секции.
  3. Не захватывайте lock дольше, чем нужно. Не делайте IO/HTTP под локом.
  4. Один объект — один подход. Не миксуйте synchronized и AtomicInteger.
  5. Тестируйте конкурентно. @RepeatedTest(100) + CountDownLatch.

10. Deadlock: бонус

// Поток A:
synchronized (book1) {
    synchronized (book2) { ... }
}

// Поток B:
synchronized (book2) {
    synchronized (book1) { ... }  // deadlock
}

Правила: всегда захватывать локи в одном порядке; лучше — не больше одного лока; jstack <pid> покажет deadlock.

# Многопоточность в Java: методичка

> Материалы для пары курса ООП. Сопровождает код `investment-app` — модуль `concurrency/`.

---

## 1. Зачем потоки вообще

Две причины писать многопоточный код:

1. **Отзывчивость UI.** Если сделать HTTP-запрос на 3 секунды в UI-потоке — окно замерзает. Решение: запустить запрос в фоне.
2. **Использовать все ядра CPU.** Современный ноутбук — 8 ядер; однопоточная программа занимает одно. Для расчётов, обработки данных, серверов под нагрузкой — важно.

В этой методичке мы сфокусируемся на **проблемах**, которые появляются, когда несколько потоков работают с **общими данными**.

## 2. Race condition: корень зла

### Учебный сценарий

В пуле лежит **10 акций**. Двое клиентов одновременно хотят купить по **7 штук**.

Наивный код:

```java
int available = 10;

public boolean buy(int qty) {
    if (available >= qty) {   // (1) проверка
        available -= qty;      // (2) действие
        return true;
    }
    return false;
}
```

В однопоточном мире работает идеально: первый клиент купит 7, второй получит отказ (осталось 3, а хочет 7).

### Что идёт не так в многопотоке

```
Момент времени | Thread A                    | Thread B
────────────────────────────────────────────────────────────────
t=1            | читает available=10         |
t=2            | проверка 10 >= 7, ок        |
t=3            |                              | читает available=10
t=4            |                              | проверка 10 >= 7, ок
t=5            | available -= 7 → 3          |
t=6            |                              | available -= 7 → -4  ←!!!
```

Оба клиента прошли проверку, оба купили. Продано **14** акций из **10**. Биржа должна доставить несуществующее — баг.

Такое называется **race condition**: результат зависит от того, как ОС расшедулила потоки. На одной машине воспроизводится часто, на другой — редко, в проде — раз в неделю в самый неподходящий момент.

### Операция «минус» — не атомарная

`available -= 7` это на самом деле три операции:
1. прочитать `available` из памяти в регистр,
2. вычесть 7,
3. записать обратно в память.

Между ними может вклиниться другой поток.

### Посмотреть на баг живьём

Код в проекте: [`UnsafeOrderBook`](../src/main/java/com/example/investment/concurrency/UnsafeOrderBook.java). Запустить можно через окно **⚔ Торговый стакан** или через тест `OrderBookConcurrencyTest`.

```
$ ./gradlew test --tests OrderBookConcurrencyTest.unsafe_statistics_overoversold
[Unsafe] прогонов=100, с переторговкой=83, макс. сверх пула=7
```

## 3. Visibility vs atomicity — две разные проблемы

Часто путают. Коротко:

### Проблема 1: atomicity (атомарность)
`i++`, `count -= n`, «прочитал-проверил-записал» — могут быть прерваны посередине. Это наш race condition.

### Проблема 2: visibility (видимость)
Поток A изменил `private boolean stopped = true`. Поток B в цикле `while (!stopped)` может никогда не увидеть это изменение — JIT-компилятор оптимизировал чтение в регистр, процессор закешировал значение.

Решение visibility — `volatile` или любой синхронизатор (`synchronized`, `Lock`, `Atomic*`).

**Наш пример затрагивает обе проблемы.** Все 4 решения ниже решают обе — поэтому `volatile` нам отдельно не нужен.

## 4. Решение №1 — `synchronized`

```java
public synchronized boolean buy(int qty) {
    if (available >= qty) {
        available -= qty;
        return true;
    }
    return false;
}
```

### Как работает

У каждого Java-объекта есть **монитор** (mutex). `synchronized` на методе = «захватить монитор `this` на время вызова». Пока один поток внутри — остальные ждут перед входом.

### Что гарантируется

1. **Взаимное исключение**: не больше одного потока одновременно в методе.
2. **Happens-before**: что поток A записал внутри блока, то поток B, войдя в этот же блок позже, обязательно увидит.

### Плюсы/минусы

✅ Просто. Нельзя забыть отпустить (JVM сделает за вас).
✅ Решает и atomicity, и visibility.
❌ Блокировка на весь метод — падает параллелизм.
❌ Нельзя прервать ожидание, нельзя с таймаутом.
❌ Нельзя использовать несколько мониторов для разных частей объекта (можно, но через `synchronized(lockObject)`).

### Синтаксис синхронизированного блока

```java
private final Object stockLock = new Object();

public boolean buy(int qty) {
    synchronized (stockLock) {   // захватываем конкретный монитор
        ...
    }
}
```

Правило: **никогда не синхронизируйтесь на `String` или боксированных `Integer`** — JVM может их интернировать, и вы случайно захватите общий монитор с другим кодом.

## 5. Решение №2 — `ReentrantLock`

```java
private final ReentrantLock lock = new ReentrantLock(true);  // fair=true

public boolean buy(int qty) {
    lock.lock();
    try {
        if (available >= qty) { available -= qty; return true; }
        return false;
    } finally {
        lock.unlock();   // ВСЕГДА в finally
    }
}
```

### Чем лучше synchronized

- **Таймаут**: `lock.tryLock(500, TimeUnit.MILLISECONDS)` — ждать не больше 500 мс.
- **Прерываемость**: `lock.lockInterruptibly()` — поток можно разбудить `interrupt()`.
- **Fairness**: `new ReentrantLock(true)` — потоки получают lock в FIFO-порядке. Справедливо, но медленнее.
- **Можно держать lock через вызовы методов** (передавать в другие классы). Для `synchronized` это неудобно.
- Несколько условий через `newCondition()` — аналог `wait/notify`, но гибче.

### Чем хуже

- **Нельзя забыть `unlock()`**. Поэтому всегда `try/finally`.
- Многословнее.

### Когда выбирать

`ReentrantLock` если:
- Нужен таймаут или прерываемость.
- Нужна fairness.
- Блокировка пересекает границы методов/классов.

Иначе — `synchronized` проще и достаточно.

## 6. Решение №3 — `AtomicInteger` + CAS

### Что такое CAS

CAS (Compare-And-Swap) — атомарная процессорная инструкция:

```
compareAndSet(expected, newValue):
    if (currentValue == expected) {
        currentValue = newValue;
        return true;
    }
    return false;
```

Выполняется за одну инструкцию. В Java доступно через `AtomicInteger`, `AtomicLong`, `AtomicReference`.

### Код

```java
private final AtomicInteger available = new AtomicInteger(10);

public boolean buy(int qty) {
    while (true) {
        int current = available.get();
        if (current < qty) return false;
        int next = current - qty;
        if (available.compareAndSet(current, next)) {
            return true;
        }
        // CAS не прошёл — другой поток успел изменить available. Ретрай.
    }
}
```

### Как это не lock?

Это **lock-free** алгоритм. Ни один поток не усыпляется — если CAS не прошёл, мы просто крутим цикл. При низкой контенции это намного быстрее `synchronized`.

### Где подводные камни

- **Проблема ABA**: значение ушло `A → B → A`. CAS этого не заметит. Для счётчика неактуально, для ссылок — надо думать (`AtomicStampedReference`).
- **Одно поле**. Если надо атомарно менять два — придётся `AtomicReference<ImmutableState>` или всё-таки lock.
- **Голодание** (starvation) при высокой контенции: CPU крутит цикл, не делая полезной работы.

### Когда выбирать

- Счётчики, флаги, simple state machines.
- Hot path — код, который гоняется миллион раз в секунду.
- При умеренной контенции.

## 7. Решение №4 — `Semaphore`

Семафор — это счётчик разрешений. Делает две операции:
- `acquire(n)` — забрать n разрешений, иначе ждать.
- `release(n)` — вернуть n разрешений.

Естественно ложится на «ограниченный ресурс»:

```java
private final Semaphore permits = new Semaphore(10, true);  // fair
private final AtomicInteger sold = new AtomicInteger();

public boolean buy(int qty) {
    if (!permits.tryAcquire(qty)) return false;
    sold.addAndGet(qty);
    return true;
}
```

### Классическое применение

- Ограничение параллельных запросов к внешнему API: `Semaphore(5)` = не больше 5 одновременно.
- Пул соединений к БД.
- Барьер «не больше N одновременных загрузок».

### Важно: семафор синхронизирует только квоту

Если у объекта есть другие поля (в нашем коде — `sold`), их надо защитить отдельно. Семафор не атомизирует всю операцию «списать разрешения и обновить статистику». Поэтому `sold` — `AtomicInteger`, иначе получим race condition второй волны.

## 8. Сравнительная таблица

| Примитив | Сложность | Производительность | Гибкость | Когда |
|----------|-----------|--------------------|----------|-------|
| `synchronized` | низкая | средняя | низкая | Простой случай, один объект, короткая критическая секция |
| `ReentrantLock` | средняя | средняя | высокая (таймаут, fairness, interruption) | Нужны нестандартные гарантии |
| `AtomicInteger / CAS` | высокая (думать о ретраях) | высокая при низкой контенции | низкая (одно значение) | Счётчик/флаг/ссылка, hot path |
| `Semaphore` | средняя | средняя | средняя | Квота ресурсов, ограничение параллелизма |

## 9. Пять правил, чтобы не наступать на грабли

1. **Не делитесь мутабельным состоянием без синхронизации.** Самый простой путь — делать immutable (`record`, `final` поля, `List.copyOf(...)`).
2. **Атомарная операция начинается и заканчивается в одной критической секции.** Если «проверить → действовать» разделены — это всегда race.
3. **Не захватывайте lock дольше, чем нужно.** Не делайте IO/HTTP под локом.
4. **Один объект — один подход.** Не миксуйте `synchronized` и `AtomicInteger` для одного и того же поля — получится странно и непредсказуемо.
5. **Тестируйте конкурентно.** `@RepeatedTest(100)` + `CountDownLatch` для синхронного старта потоков. Баги в потоках не ловятся линтерами.

## 10. Deadlock: бонус-раздел

Взяли два лока в разном порядке — приехали.

```java
// Поток A:
synchronized (book1) {
    synchronized (book2) { ... }
}

// Поток B:
synchronized (book2) {   // ← уже держит book2
    synchronized (book1) { ... }  // ← ждёт book1, который держит A
}
// Оба стоят. Deadlock.
```

Правила:
- Всегда захватывать локи в одном и том же порядке.
- Лучше — не захватывать больше одного лока одновременно.
- `jstack <pid>` покажет «Found 1 deadlock» и стек обоих потоков.
